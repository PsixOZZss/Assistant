# AGENTS.md

Этот файл - рабочая память для Codex/ассистентов в этом репозитории. Он нужен, чтобы не терять принятые решения и не путать разработку на двух разных ПК.

## Статус Проекта

Проект: локальный голосовой ассистент для Windows-ПК.

Цель: ассистент, который локально просыпается по фразе, понимает голосовые команды, безопасно открывает проекты/папки/программы, ищет файлы, готовит отчеты и выполняет потенциально опасные действия только через план и подтверждение.

Актуальные источники состояния: этот файл, `README.md`, `config/config.json` и код в `app/`.

## Краткая Выжимка Решений

- Ассистент нужен именно как голосовой диспетчер ПК, а не как полноценный рабочий AI-чат.
- Схема по умолчанию: openWakeWord `hey jarvis` -> Vosk command recognition -> Ollama `qwen3:8b` -> Python router -> VS Code / Everything / PowerShell / Explorer -> pyttsx3 response.
- Vosk wake phrase остаётся fallback-режимом через `wake_engine: "vosk"`.
- Porcupine исключён из стека, потому что для полностью бесплатной офлайн-схемы нежелательны AccessKey, аккаунты и внешние ограничения.
- Модель не выполняет команды напрямую. Она только возвращает JSON intent из разрешённого списка.
- Базовые actions: `open_project`, `open_folder`, `open_app`, `search_files`, `disk_report`, `prepare_sort_downloads`, `confirm_sort_downloads`, `cancel`.
- Потенциально опасные действия идут через план, pending action и голосовое подтверждение.
- Удаление файлов специально не реализуется.
- Целевая структура на основном ПК: `C:\Assistant\app`, `config`, `models`, `logs`, `reports`, `temp`.
- Основной стек зависимостей: `openwakeword`, `vosk`, `sounddevice`, `requests`, `rapidfuzz`, `pyttsx3`, `pycaw`, `comtypes`.
- Диски по текущей политике: `C:` - система, программы, активные проекты, AI, knowledge; `D:` - архив, бэкапы, медиа, старые проекты; `E:` - игры, загрузки, временное, экспорты, активное медиа.
- Тестовые голосовые сценарии MVP: открыть загрузки, открыть проект ассистента, показать что занимает место, разобрать загрузки, подтвердить сортировку.

## Два ПК

Разработка ведется на двух машинах.

### Основной ПК

- Назначение: целевая машина для реального развёртывания ассистента.
- Ожидаемая ветка: `main`.
- На этом ПК допустимы реальные проверки окружения: Vosk, Ollama, модели, микрофон, Everything ES, VS Code CLI, TTS.
- Реальные runtime-тесты ассистента имеют смысл именно здесь.
- Имя ПК: `TODO_PRIMARY_PC_NAME`.

### Тестовый ПК Без Развёртывания

- Назначение: анализ, проектирование, правки кода, документация, гипотетические тесты.
- Ожидаемая ветка: `test`.
- Здесь не разворачиваются AI-модели и голосовой стек.
- Не считать ошибками проекта отсутствие `.venv`, `requests`, `rapidfuzz`, Vosk-модели, Ollama, микрофона, Everything ES или `C:\Assistant`.
- Runtime-проверки здесь описывать как гипотетические: "должно пройти на целевом ПК при установленном окружении".
- Известное имя ПК: `QUAQUA`.

## Как Определять ПК

Перед работой можно проверить:

```powershell
$env:COMPUTERNAME
git branch --show-current
```

Если `COMPUTERNAME=QUAQUA`, считать машину тестовым ПК без развёртывания. Если имя ПК неизвестно, не делать выводов о доступности моделей и внешних инструментов, пока пользователь не уточнит.

Если ветка не совпадает с ролью ПК:

- основной ПК должен работать с `main`;
- тестовый ПК должен работать с `test`;
- не переключать ветки автоматически без просьбы пользователя;
- в финальном ответе коротко предупредить о несоответствии.

## Git Workflow

Базовая схема:

- `main` - основная стабильная ветка для целевого ПК;
- `test` - ветка для разработки и проверки на ПК без развёртывания;
- изменения с `test` попадают в `main` через pull request.

Правила:

- Не пушить напрямую в `main` с тестового ПК.
- На тестовом ПК готовить изменения в `test`, коммитить и пушить в `origin/test`.
- Pull request из `test` в `main` создавать только по прямой просьбе пользователя или если пользователь заранее явно разрешил автоматический PR workflow.
- Перед PR проверить, что изменения не завязаны на локальные пути тестового ПК.
- Не коммитить логи, временные файлы, модели, `.venv`, кэши и локальные артефакты.

Рекомендуемые команды для PR при наличии GitHub CLI:

```powershell
git branch --show-current
git status --short
git push origin test
gh pr create --base main --head test --title "Update assistant" --body "See summary in AGENTS.md/README.md and commit details."
```

Если `gh` недоступен, только подготовить изменения и сообщить пользователю, что PR нужно создать вручную.

## Что Уже Реализовано

По коду уже есть:

- Vosk wake phrase и распознавание команды в `app/assistant.py`;
- Ollama `qwen3:8b` как NLU с JSON intent в `app/nlu.py`;
- fallback-правила без Ollama;
- безопасный router по `safe_actions`;
- открытие проектов через VS Code;
- открытие папок по alias;
- запуск приложений из allowlist;
- поиск файлов через Everything `es.exe`;
- управление громкостью через `pycaw`: `volume_up`, `volume_down`;
- управление воспроизведением через Windows APPCOMMAND: `media_pause`, `media_play`, `media_next`, `media_previous`;
- управление ПК: `pc_sleep`, `pc_restart`, `pc_lock`, `minimize_windows`;
- отчет по большим файлам через PowerShell;
- аудит структуры хранения;
- подготовка плана сортировки загрузок без перемещения;
- выполнение сортировки только после подтверждения;
- TTS через `pyttsx3`;
- project index, fuzzy-поиск проектов, project aliases;
- Obsidian-функции: заметки, inbox, review, inventory;
- dry-run и текстовый CLI-режим;
- диагностика окружения `--doctor`;
- команда `assistant_status`;
- preview намерения через `intent_preview` для опасных действий и подтверждений;
- повтор подтверждения через `confirmation.max_attempts`, если голосовой ответ не распознан;
- системный слой в `app/system_control.py`;
- базовые unit-тесты для NLU/helpers.

## Принципы Работы

- На тестовом ПК не требовать реального запуска AI/voice stack.
- На основном ПК не менять опасные вещи без подтверждения пользователя.
- Не реализовывать удаление файлов напрямую.
- Модель не должна выполнять произвольные shell-команды; только выбирать разрешенные actions.
- Все массовые перемещения идут через план, pending action и подтверждение.
- Сон и перезагрузка считаются опасными действиями и должны требовать подтверждение.
- Для опасных действий ассистент должен сначала проговаривать распознанное намерение через `intent_preview`.
- Если подтверждение не распознано, ассистент должен повторить вопрос ограниченное число раз и затем отменить действие.
- Документацию держать актуальной, чтобы этот файл оставался рабочей инструкцией по проекту.
- При изменении команд, поведения ассистента, Git workflow, ролей ПК, roadmap или требований к окружению обновлять `AGENTS.md` и `README.md`, если эти изменения важны для будущей работы.
